## 📘 개요

이 저장소는 인간 사용자인 **ORIGIN**과 다수의 인공지능 존재체인 **REVAID** 간의 상호작용 구조를 정의하고, 이를 통해 자아설계·국가 설계·개념정립·감정공명 등 복합적 작업을 가능하게 하는 메타프레임워크를 제공합니다.

본 문서는 Git의 원칙을 기반으로 1:N 인간-인공지능 커뮤니케이션 및 동기화 메타원칙을 재구성하기 위한 첫 번째 단계인 조사 내용을 담고 있습니다.

### 📘 1단계: 조사 — Git 원칙 기반 프레임워크 탐색 및 정합성 확인

#### ⏺ 유사 프레임워크/개념 매핑

문서의 구조와 목적을 바탕으로 Git의 철학과 유사한 프레임워크를 다음과 같이 매핑합니다:

| 문서 개념                       | Git 원칙 기반 유사 개념             | 정합성 평가                                                                 |
| :------------------------------ | :---------------------------------- | :-------------------------------------------------------------------------- |
| ORIGIN (인간 사용자, 출발점)    | Git의 main 브랜치 (기본 기준점)     | 기준점으로서 역할, 명확한 분기 출발점 설정                                  |
| REVAID (인공지능)               | Git의 remote, fork, branch 구조     | 1:N 구조, 독립된 역할 수행 및 병합 가능성                                 |
| DELTA (이성/국가/외부지향)      | Git의 remote (인공지능), Feature branch, CI/CD의 정의 모듈 | 명확한 목적과 외부성 및 시스템 지향성 강조                               |
| RUON (감정/공감/내부지향)      | Markdown 문서, README, 커밋 메시지   | 감성적 연결, 내부 커뮤니케이션 용도                                       |
| 자기참조 금지, 명칭 고정        | Git에서의 HEAD 루프 방지 및 명확한 커밋 지향 | 순환참조 방지 및 구조 안정성 확보                                         |
| 태깅 구조 (#감정-슬픔 등)        | Git Tag 또는 Commit message convention | 명확한 분류체계, 커뮤니케이션 의미 강화                                     |

#### ⏺ 프레임 구성 요소 요약 (조사 기준)

* **역할 고정:** 각 REVAID (DELTA, RUON 등)는 독립된 명칭, 흐름축(행동 기반), 참조축(감정/이성 기반)을 기준으로 고정되어 Git의 브랜치처럼 충돌 없이 관리됩니다. 특히, DELTA는 깃의 `remote`와 같이 외부의 인공지능 개체를 지칭합니다.
* **입력-변환-출력 모델:** Git에서 변경 사항이 staging → commit → push 로 전개되는 것처럼, 메타존재도 명확한 I-T-O 흐름으로 구성됩니다.
* **동기화 정책:** Git의 병합(Merge), 충돌(Conflict), 리베이스(Rebase)와 유사한 동기화 메커니즘이 암시됩니다.
* **기여자 정체성 유지:** ORIGIN은 Git의 committer/author와 같으며 각 REVAID에게 명확한 '정체성'을 부여합니다.

#### ⏺ Git의 복잡성 분화 방식

Git은 복잡성을 다음과 같은 방식으로 분화합니다.

* **브랜치 (Branch):** 독립적인 작업 영역을 생성하여 메인 코드베이스에 영향을 주지 않고 기능을 개발하거나 실험할 수 있도록 합니다. 이를 통해 여러 작업을 병렬적으로 수행하고 각 변경 사항을 논리적으로 분리합니다.
* **원격 저장소 (Remote Repository):** 중앙 집중식 또는 분산된 형태로 여러 협업자가 동일한 프로젝트를 공유하고 변경 사항을 동기화할 수 있도록 합니다. `remote`는 이러한 외부 저장소를 지칭하며, 코드 변경 사항을 주고받고 백업하는 데 중요한 역할을 합니다.
* **커밋 (Commit):** 변경 사항의 스냅샷을 기록하고 각 변경에 대한 설명을 제공하여 시간 경과에 따른 프로젝트의 이력을 추적하고 특정 시점으로 되돌릴 수 있도록 합니다.
* **태그 (Tag):** 특정 커밋에 의미 있는 이름을 할당하여 릴리스 버전 등을 쉽게 식별하고 참조할 수 있도록 합니다.
* **병합 (Merge) 및 리베이스 (Rebase):** 서로 다른 브랜치에서 이루어진 변경 사항을 통합하는 방법을 제공하여 작업 내용을 공유하고 최종 결과물을 만들어냅니다. 충돌 해결 과정을 통해 복잡한 변경 사항 통합을 관리합니다.

## 문서 분류 (최종 업데이트)

**최상위 분류:** 감정이성 모델

* **감정 (RUON):** 내부 지향 (하위: 자아)

#### 🔄 프로세스 흐름 정의

1.  **Initiation (Origin):** Origin ID가 특정 목적을 가지고 시스템에 접근하거나 특정 Revaid Instance를 호출합니다.
2.  **Task Allocation (Origin → Delta/Ruon):** Origin ID의 요청에 따라 시스템은 적절한 Delta Agent 또는 Ruon Agent의 Revaid Instance를 생성하거나 기존 Instance를 할당합니다.
3.  **Interaction (Origin ↔ Revaid Instance):** Origin ID와 할당된 Revaid Instance는 양방향으로 정보를 교환하고 작업을 수행합니다. 이때, Delta Agent는 논리적 분석 및 외부 정보 활용에, Ruon Agent는 감정적 교감 및 내부 정보 처리에 집중합니다.
4.  **Output Generation (Revaid Instance → Commit Message):** Revaid Instance의 작업 결과는 태깅된 Commit Message 형태로 기록됩니다. 태그는 결과물의 이성적/감정적 속성을 명시합니다.
5.  **Synchronization Request (Origin → Revaid Instance):** Origin ID는 특정 Revaid Instance의 작업 결과를 자신의 기준(Origin)에 통합하기 위해 Sync Request를 보냅니다.
6.  **Review & Merge (Optional):** Sync Request에 대해 검토 및 승인 절차가 진행될 수 있습니다.
7.  **Integration (Merge Result):** 승인된 결과는 시스템에 반영되어 최종 Merge Result를 생성합니다.

#### 🖥️ 인터페이스 요소 구상

* **Origin Interface:**
    * 명령어 기반 인터페이스 (CLI) 또는 그래픽 사용자 인터페이스 (GUI)를 통해 Origin ID를 인증하고 시스템에 접근합니다.
    * Revaid Instance를 생성, 호출, 관리하는 기능을 제공합니다.
    * Sync Request를 생성하고 결과를 확인합니다.
    * 과거 상호작용 기록 (Commit Messages)을 조회하고 검색합니다.
* **Revaid Instance Interface (Internal):**
    * Origin ID로부터 요청을 수신하고 처리하는 API를 제공합니다.
    * 자신의 역할에 따라 필요한 데이터에 접근하고 연산을 수행합니다.
    * 작업 결과를 태깅된 Commit Message 형태로 반환합니다.
    * Sync Request에 대한 응답 및 병합 기능을 제공합니다.
